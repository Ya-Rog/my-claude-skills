# Skill Creator — создание и улучшение скилов

Скил для создания новых скилов и их итеративного улучшения.

## Процесс создания скила

На высоком уровне процесс выглядит так:

1. Определи что должен делать скил и как примерно он должен это делать
2. Напиши черновик скила
3. Создай несколько тестовых промптов и запусти claude-with-access-to-the-skill на них
4. Помоги пользователю оценить результаты качественно и количественно
5. Перепиши скил на основе обратной связи
6. Повтори до удовлетворения

Твоя задача — определить где пользователь находится в этом процессе и помочь ему продвинуться через эти этапы.

## Общение с пользователем

Обращай внимание на контекстные подсказки о техническом уровне пользователя. По умолчанию:
- "оценка" и "бенчмарк" — нормальные термины
- "JSON" и "assertion" — объясняй если не уверен что пользователь знает что это

## Создание скила

### Захват намерения

Пойми намерение пользователя. Если текущий разговор уже содержит workflow — сначала извлеки ответы из истории переписки.

1. Что должен уметь делать этот скил?
2. Когда должен срабатывать? (какие фразы/контексты)
3. Какой ожидаемый формат вывода?
4. Нужны ли тест-кейсы для проверки? (скилы с объективно проверяемыми выводами — файловые трансформации, извлечение данных — выигрывают от тест-кейсов)

### Написание SKILL.md

Заполни компоненты:

- **name**: идентификатор скила (только буквы, цифры, дефисы)
- **description**: когда срабатывать, что делает. Это основной механизм срабатывания — включай и что делает скил И конкретные контексты когда использовать. Делай описания немного "настойчивыми" — Claude склонен к under-triggering.
- **тело скила**: инструкции

### Структура SKILL.md

```
skills/skill-name/
├── SKILL.md (обязательно)
│   ├── YAML frontmatter (name, description обязательны)
│   └── Markdown инструкции
└── Ресурсы (опционально)
    ├── scripts/    — исполняемый код для повторяющихся задач
    ├── references/ — документация загружаемая по необходимости
    └── assets/     — файлы используемые в выводе
```

**Progressive Disclosure — три уровня загрузки:**
1. **Метаданные** (name + description) — всегда в контексте (~100 слов)
2. **Тело SKILL.md** — в контексте когда скил срабатывает (<500 строк идеально)
3. **Ресурсы** — по необходимости (без ограничений)

### Принципы написания

- Предпочитай повелительную форму в инструкциях
- Объясняй **почему** важны те или иные вещи, а не просто диктуй правила
- Старайся сделать скил общим, не узким под конкретные примеры
- Если пишешь ALWAYS или NEVER заглавными — это жёлтый флаг: лучше объясни причину

### Описание (CSO — Claude Search Optimization)

**Описание = Когда использовать, НЕ что делает скил**

```yaml
# ❌ ПЛОХО: описывает процесс — Claude может следовать ему вместо чтения скила
description: Используй при выполнении планов — создаёт субагента на задачу с ревью между ними

# ✅ ХОРОШО: только условия срабатывания
description: Используй при выполнении планов реализации с независимыми задачами
```

## Запуск и оценка тест-кейсов

### Шаг 1: Запустить все прогоны (с скилом И без) в одном ходу

Для каждого тест-кейса запускай два субагента одновременно — один с скилом, один без. Важно: не запускай сначала with-skill а потом возвращайся за baseline — запускай всё сразу.

### Шаг 2: Пока прогоны идут — пиши assertions

Не жди завершения — используй время продуктивно. Пиши количественные assertions для каждого тест-кейса и объясняй их пользователю.

### Шаг 3: Когда прогоны завершаются — захвати timing данные

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

### Шаг 4: Оцени, агрегируй и запусти просмотрщик

```bash
python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
nohup python <skill-creator-path>/eval-viewer/generate_review.py \
  <workspace>/iteration-N --skill-name "my-skill" > /dev/null 2>&1 &
```

**ВАЖНО**: Генерируй eval-просмотрщик ПЕРЕД самостоятельной оценкой. Хочешь показать результаты пользователю как можно раньше.

## Улучшение скила

### Как думать об улучшениях

1. **Обобщай из обратной связи** — скил будет использоваться миллион раз; не оптимизируй только под текущие тест-кейсы
2. **Держи промпт компактным** — убирай то что не работает; читай транскрипты, не только финальные выводы
3. **Объясняй почему** — сегодняшние LLM умны; объяснение причин эффективнее жёстких правил
4. **Ищи повторяющийся код** — если все 3 тест-кейса написали похожий helper-скрипт → помести его в `scripts/`

## Оптимизация описания

После создания/улучшения скила предложи оптимизировать описание для лучшей точности срабатывания:

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id> \
  --max-iterations 5 \
  --verbose
```

Скрипт автоматически:
- Делит eval-сет на 60% train / 40% test
- Оценивает текущее описание (каждый запрос x3 для надёжности)
- Предлагает улучшения через Claude с extended thinking
- Возвращает `best_description` — по test score, не train, чтобы избежать overfitting
